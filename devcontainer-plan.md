# DevContainer Plan for ModusToolbox + mcu-debug

## Context

- **Repo**: `Hello_World` — PSoC6 HAL example (CY8CKIT-062-WIFI-BT), uses `cortex-debug` extension
- **Goal**: Containerized build environment with ModusToolbox 3.7 tools, reusable across multiple test-case repos, usable on macOS and Windows
- **Secondary goal**: Debug the `mcu-debug` extension itself inside the container
- **Base Dockerfile**: `/Users/hdm/mtb-docker-tools-main/Dockerfile_20260326_130143` — auto-generated by `generate_dockerfile.py`, installs GCC ARM 14.2.1, MTB Tools 3.7, OpenOCD (ProgTools 1.7), ZAP, Edge Protect, CAPSENSE/Multi-Sense Pack, ML Pack, Industrial MCU Pack, Audio SW Codecs on Ubuntu 22.04

## 1. How DevContainers Work (Quick Primer)

When you open a folder in a devcontainer:

```
┌─────────────────────────────────┐    ┌──────────────────────────────────────┐
│  YOUR MAC / WINDOWS             │    │  DOCKER CONTAINER                    │
│                                 │    │                                      │
│  VS Code UI (Electron)          │◄──►│  VS Code Server (headless)           │
│  UI Extensions (local)          │    │  Workspace Extensions (remote)       │
│  USB devices (probe)            │    │  GCC ARM, Make, OpenOCD, etc.        │
│                                 │    │  Your source code (bind-mounted)     │
└─────────────────────────────────┘    └──────────────────────────────────────┘
```

- **VS Code itself does NOT need to be installed in the image**. VS Code automatically injects its server into the container at launch.
- **Node.js**: VS Code Server bundles its own Node. You only need Node in the image if you're *developing/debugging* a VS Code extension inside the container (see Section 6).
- Source code is bind-mounted from host → container. Changes are live on both sides.

## 2. Architecture: Two-Layer Image Strategy (IMPLEMENTED)

### Layer 1: Base MTB Image — `mtb-tools:3.7-base`

The auto-generated Dockerfile from `mtb-docker-tools-main`, used **as-is** with no modifications. Contains:

- Ubuntu 22.04
- GCC ARM toolchain 14.2.1.265 → `/opt/Tools/mtb-gcc-arm-eabi/14.2.1/gcc/bin/`
- ModusToolbox Tools 3.7.0.18135 → `/opt/Tools/ModusToolbox/tools_3.7/`
- OpenOCD (ModusToolboxProgtools 1.7.0.1727) → `/opt/Tools/ModusToolboxProgtools-1.7/openocd/bin/`
- Edge Protect Security Suite 1.6.1
- ZAP tool 2025.5.14
- CAPSENSE/Multi-Sense Pack 1.3.0
- Machine Learning Pack 3.1.0
- Industrial MCU Pack 2.1.0
- Audio SW Codecs Pack 1.0.2
- All system dependencies (Qt6, libusb, locales, git, python3-venv, etc.)
- Locale set to `en_US.UTF-8`

**Actual size**: ~8.35 GB

**Build command:**
```bash
docker build --platform linux/amd64 \
  -f /Users/hdm/mtb-docker-tools-main/Dockerfile_20260326_130143 \
  -t mtb-tools:3.7-base \
  /Users/hdm/mtb-docker-tools-main
```

### Layer 2: DevContainer Layer — `.devcontainer/Dockerfile` (per-repo, thin)

A small Dockerfile in each repo's `.devcontainer/` that does `FROM mtb-tools:3.7-base` and adds only:

- Non-root `vscode` user (UID 1000) with passwordless sudo
- `/workspaces/mtb_shared` directory with correct ownership
- PATH additions for MTB tools, GCC ARM, and OpenOCD

**Why two layers?**
- The generated Dockerfile changes when tools are updated — no manual merging needed
- The devcontainer layer is tiny and fast to rebuild
- Separation of concerns: Infineon tools vs. VS Code integration

**Current `.devcontainer/Dockerfile`:**
```dockerfile
FROM mtb-tools:3.7-base

ARG USERNAME=vscode
ARG USER_UID=1000
ARG USER_GID=${USER_UID}
RUN groupadd --gid ${USER_GID} ${USERNAME} \
    && useradd --uid ${USER_UID} --gid ${USER_GID} -m ${USERNAME} -s /bin/bash \
    && echo "${USERNAME} ALL=(ALL) NOPASSWD:ALL" > /etc/sudoers.d/${USERNAME} \
    && chmod 0440 /etc/sudoers.d/${USERNAME}

RUN mkdir -p /workspaces/mtb_shared \
    && chown ${USERNAME}:${USERNAME} /workspaces/mtb_shared

ENV PATH="/opt/Tools/ModusToolbox/tools_3.7/bin:\
/opt/Tools/mtb-gcc-arm-eabi/14.2.1/gcc/bin:\
/opt/Tools/ModusToolboxProgtools-1.7/openocd/bin:\
${PATH}"
```

## 3. Per-Repo DevContainer Config (IMPLEMENTED)

### `.devcontainer/devcontainer.json`

```jsonc
{
    "name": "MTB Hello World",
    "build": {
        "dockerfile": "Dockerfile",
        "options": ["--platform=linux/amd64"]
    },
    "runArgs": ["--platform=linux/amd64"],
    "customizations": {
        "vscode": {
            "extensions": [
                "ms-vscode.cpptools",
                "dan-c-underwood.arm",
                "llvm-vs-code-extensions.vscode-clangd",
                "mcu-debug.debug-tracker-vscode",
                "mcu-debug.memory-view",
                "mcu-debug.rtos-views",
                "mcu-debug.peripheral-viewer",
                "marus25.cortex-debug"
            ],
            "settings": {
                "terminal.integrated.defaultProfile.linux": "bash",
                "modustoolbox.toolsPath": "/opt/Tools/ModusToolbox/tools_3.7",
                "cortex-debug.armToolchainPath": "/opt/Tools/mtb-gcc-arm-eabi/14.2.1/gcc/bin",
                "cortex-debug.openocdPath": "/opt/Tools/ModusToolboxProgtools-1.7/openocd/bin/openocd",
                "mcu-debug.armToolchainPath": "/opt/Tools/mtb-gcc-arm-eabi/14.2.1/gcc/bin",
                "mcu-debug.openocdPath": "/opt/Tools/ModusToolboxProgtools-1.7/openocd/bin/openocd"
            }
        }
    },
    "remoteUser": "vscode",
    "workspaceFolder": "/workspaces/${localWorkspaceFolderBasename}",
    "postCreateCommand": "make getlibs && make vscode && make build -j$(nproc)"
}
```

**Notes:**
- `--platform=linux/amd64` is required because Infineon tools are x86_64-only (runs under Rosetta on Apple Silicon Macs)
- VS Code extension dependencies are NOT auto-installed — all required extensions must be listed explicitly
- Settings in `customizations.vscode.settings` override workspace `.vscode/settings.json`, so macOS-native paths in the repo are untouched
- `postCreateCommand` runs on first container creation: downloads libraries, generates VS Code project files, and verifies the build

### `.gitattributes` (for cross-platform line endings)

```
* text=auto eol=lf
```

### mtb_shared

The MTB build system expects shared libraries at `../mtb_shared/` relative to the project. Since the workspace mounts at `/workspaces/Hello_World`, this resolves to `/workspaces/mtb_shared/`.

**Solution (implemented):** Pre-create `/workspaces/mtb_shared` in the Dockerfile with `vscode` ownership. `make getlibs` populates it on first run via `postCreateCommand`. This is better than bind-mounting from the host because it keeps the container self-contained.

## 4. Reusing the Image Across Multiple Repos

**Yes — this is the entire point of the two-layer strategy.**

Each test-case repo gets only a `.devcontainer/` directory. They all share the same `mtb-tools:3.7-base` image.

```
repo-1/.devcontainer/  →  FROM mtb-tools:3.7-base  (same base)
repo-2/.devcontainer/  →  FROM mtb-tools:3.7-base  (same base)
repo-3/.devcontainer/  →  FROM mtb-tools:3.7-base  (same base)
```

**What varies per repo:**
- Extension list (some repos may not need all extensions)
- `customizations.vscode.settings` (different target boards, toolchain overrides)
- `postCreateCommand` (repo-specific setup)

**Note:** Each repo gets its own container instance (with its own `/workspaces/mtb_shared`), but they all share the cached base image layers — no duplicate disk usage for the toolchains.

## 5. Docker Image Storage & Sharing

### Registry Options (recommended over local SSD)

| Method | Command | Cost | Best for |
|--------|---------|------|----------|
| Docker Hub (free) | `docker push yourdockerhubuser/mtb-tools:3.7-base` | Free (1 private repo, unlimited public) | Public sharing |
| GitHub Container Registry | `docker push ghcr.io/mcu-debug/mtb-tools:3.7-base` | Free for public repos | Keeping it with mcu-debug org |
| Docker Hub (Pro) | Same as above | $7/mo | Multiple private repos |
| Export to tarball | `docker save mtb-tools:3.7-base -o mtb-tools.tar` | Free | Sneakernet / offline backup |

### Push to GHCR example:
```bash
# Login (uses your GitHub PAT with packages:write scope)
echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin

# Tag and push
docker tag mtb-tools:3.7-base ghcr.io/mcu-debug/mtb-tools:3.7-base
docker push ghcr.io/mcu-debug/mtb-tools:3.7-base

# Then in devcontainer Dockerfile, change FROM to:
# FROM ghcr.io/mcu-debug/mtb-tools:3.7-base
```

### Local storage on external SSD (`/Volumes/VMs`)

If you prefer local storage, Docker Desktop stores everything in a single VM disk image:
```
~/Library/Containers/com.docker.docker/Data/vms/0/data/Docker.raw
```

**To move it:**
Docker Desktop → Settings → Resources → Advanced → **Disk image location** → `/Volumes/VMs/docker-data/`

**Caveats:**
- SSD must be mounted when Docker runs
- Must be APFS or HFS+ (not exFAT/FAT32)
- USB-C/Thunderbolt performance is fine; USB-A 3.0 will be noticeably slower

### Docker Dashboard `<none>` tags

The `vsc-*` images shown as `<none>` in Docker Dashboard are auto-generated by VS Code's devcontainer infrastructure. They are normal and expected — VS Code creates them when building the devcontainer layer. Our actual images (`mtb-tools:3.7-base`, `mtb-tools:3.7`) are correctly tagged.

## 6. Do You Need Node.js / VS Code in the Image?

### Short answer: **Not for normal use. Only for extension debugging.**

### Scenario A: Building firmware and debugging on-target
- **Node.js**: Not needed. VS Code Server bundles its own.
- **VS Code**: Not installed in image. VS Code injects its server automatically.
- **This is the current setup — it works.**

### Scenario B: Debugging the `mcu-debug` extension itself

When you do F5 "Run Extension" on your extension source, VS Code launches an Extension Development Host. In a devcontainer, you'd need Node.js for `npm install` / `npm run compile`.

**Lightweight approach — add a devcontainer Feature (no base image change):**
```jsonc
// In devcontainer.json for extension dev repos only:
{
    "features": {
        "ghcr.io/devcontainers/features/node:1": {
            "version": "20"
        }
    }
}
```

This adds ~100 MB on top. The base image stays clean. Only add this to repos where you're actively developing the extension.

## 7. Windows Portability

### Will the same Docker image work on Windows?

**Yes.** Docker images are Linux containers. On Windows they run under WSL2. The same `mtb-tools:3.7-base` image works unchanged.

### What's different on Windows?

| Aspect | macOS | Windows |
|--------|-------|---------|
| Docker runtime | Docker Desktop (Apple VZ framework) | Docker Desktop (WSL2 backend) |
| USB passthrough | Not supported (Docker runs in a VM) | Via `usbipd-win` into WSL2 |
| Filesystem perf | Good with VirtioFS mounts | Poor with `/mnt/c/...`; good with WSL2-native paths |
| Image storage | `~/Library/Containers/...` or custom | `\\wsl$\docker-desktop-data\...` |
| SSD storage | Move Docker.raw (see Section 5) | `wsl --export` / `wsl --import` to SSD |

### Windows-specific gotchas

1. **File performance**: Store repos inside WSL2's filesystem (`/home/user/...`), NOT on `/mnt/c/`. NTFS→ext4 translation is very slow for builds with many small files.
2. **USB debugging**: Install `usbipd-win` to attach USB debug probes into WSL2/container.
3. **Line endings**: Already handled by `.gitattributes` with `* text=auto eol=lf`.

## 8. USB Debug Probe Access from Container

This is the trickiest part and directly ties into the mcu-debug Proxy Plan.

| Platform | USB Passthrough | Status |
|----------|----------------|--------|
| Linux (native Docker) | `docker run --device=/dev/ttyACM0` | Works well |
| macOS (Docker Desktop) | Not supported (VM barrier) | Need proxy/network approach |
| Windows (WSL2) | `usbipd-win` | Works, some friction |

### Options for macOS

1. **Proxy approach (Topology A from Proxy Plan)**: Run OpenOCD/GDB server on host, DA in container connects via TCP. Best long-term solution — this is what mcu-debug-proxy is designed for.

2. **OpenOCD on host, GDB in container**: Run `openocd` natively on macOS. Container's DA connects to `host.docker.internal:3333` (OpenOCD's GDB port).

3. **Hybrid (recommended for now)**: Use the container only for building. Debug natively on macOS. Gets 80% of the benefit immediately.

## 9. Known Issues & Gotchas

### MTB Build System vs. Git Worktrees (BUG)

The MTB build system uses wildcard globbing (e.g., `$(wildcard **/*.c)`) that recurses into `.claude/worktrees/` and other dot-directories. This causes multiply-defined symbol errors when a worktree contains copies of source files.

**Status**: This is a bug in the MTB build system. It should not descend into directories not specified in the Makefile. Worktrees are a standard git feature.

**Workaround**: Don't use git worktrees inside MTB repos, or ensure worktrees are outside the repo directory.

### Rosetta / x86_64 Emulation

All Infineon tools are x86_64 Linux binaries. On Apple Silicon Macs, they run under Rosetta emulation inside Docker's `linux/amd64` container. This works but adds some build time overhead (~1.5-2x slower than native).

### Extension Dependencies

VS Code does not auto-install extension dependencies when installing extensions in a container. All required extensions must be explicitly listed in `devcontainer.json`.
